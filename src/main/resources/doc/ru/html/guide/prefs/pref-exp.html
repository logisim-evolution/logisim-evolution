Вкладка «Экспериментальные»

На этой вкладке находятся параметры, которые считаются экспериментальными. Они добавлены для сбора отзывов от пользователей с целью дальнейшего улучшения программы.

  • Ускорение графики: использование аппаратного ускорения графики может повысить производительность отрисовки Logisim-evolution (например, с параметром командной строки -Dsun.java2d.d3d=True). Этот выпадающий список позволяет выбрать тип графического ускорения (Direct3D, OpenGL, Metal и др.).
    Изменение вступит в силу только после перезапуска программы.

  • Очередь моделирования (Simulation Event Queue): это приоритетная очередь событий, которые обрабатываются ядром моделирования. Выбор её реализации напрямую влияет на производительность и скорость моделирования схем. Наиболее эффективный алгоритм для одного проекта может оказаться не самым быстрым для другого.

    В настоящее время предлагается 5 вариантов реализации. Разработчики будут рады получить отзывы о том, какой из них работает быстрее на ваших схемах (тема обсуждения «Simulation Queue» на GitHub: https://github.com/logisim-evolution/logisim-evolution/discussions).
    Изменение вступит в силу только после закрытия и повторного открытия файла проекта.

    • Приоритетная очередь Java (Java Priority Queue): используется по умолчанию. Надёжный и стабильный стандартный вариант, однако не самый производительный на сложных схемах.
    • Приоритетная очередь на базе Splay-дерева (Splay Tree Priority Queue): может работать быстрее, но на некоторых проектах дерево может оказаться несбалансированным. Так как этот алгоритм использует рекурсию, при очень большом количестве событий может произойти переполнение стека (stack overflow).
    • Связная приоритетная очередь (Linked Priority Queue): связный список, при вставке выполняющий поиск с конца. Работает быстро для некоторых проектов, но имеет низкую производительность в худшем случае.
    • Сортированный связный список очередей (Sorted Linked List of Queues): может быть очень быстрым. Использует упорядоченный список временных узлов, каждый из которых содержит стандартную связную очередь для всех событий, запланированных на это время.
    • Сортированное дерево очередей (Sorted Tree of Queues): также может работать очень быстро. Использует отсортированное дерево для временных узлов. Работает лучше, чем список очередей, в проектах с большим количеством разнообразных задержек, но создаёт накладные расходы на простых схемах. На очень крупных проектах может работать медленнее, чем сортированный связный список очередей.
